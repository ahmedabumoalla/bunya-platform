وُحّدت هندسة اللوحات عبر Sidebar بعرض مدمج، وHeader وBreadcrumb أكثر هدوءًا، ومساحات محتوى محكومة، وتسلسل ثابت للعناوين والبطاقات والمؤشرات. وُحّدت البطاقات والنماذج والجداول والفلاتر والشارات وحالات الفراغ والتحميل والتنبيه والنوافذ، وأصبحت الجداول ذات Header ثابت وتمرير أفقي داخلي منظم. عولج تجاوز عرض نموذج رفع منتج المزود، وأصبحت شبكات النماذج والقياسات والصور تتكيف بلا overflow على الجوال واللوحي والمكتبي.

أضيف `src/components/BunyaVisualRuntime.tsx` كسلوك وصول بصري مشترك للنوافذ الموجودة: ينقل التركيز إلى النافذة، يحصر التنقل بـTab داخلها، يغلقها بزر Escape عبر إجراء الإغلاق الموجود، يعيد التركيز إلى العنصر السابق، ويمنح أزرار الإغلاق الرمزية تسمية عربية. أضيفت حركات دخول قصيرة تعتمد على opacity وtransform مع احترام `prefers-reduced-motion`، وخُفّض استخدام `backdrop-filter` في البطاقات المتكررة لدعم الأداء.

التعديل Presentation-only: لم تتغير Routes أو Auth أو Route Guards أو مفاتيح localStorage أو البيانات أو الحالات أو API أو PWA أو Migration 001، ولم تُضف مكتبات.

## بوابات الدخول والانضمام — ملف 008

### Routes الجديدة

- `/login`: تسجيل دخول عميل بمحاكاة محلية.
- `/register`: إنشاء حساب عميل محلي.
- `/forgot-password`: تسجيل طلب استعادة محلي دون إرسال فعلي.
- `/reset-password`: واجهة تعيين كلمة جديدة، وتقرأ وجود `token` مستقبلًا من Query Parameter دون التحقق منه حاليًا.
- `/providers/join`: طلب انضمام مزود.
- `/contractors/join`: طلب انضمام مقاول.

صفحة `/contractors` كانت موجودة، وتم تحويلها إلى دليل تفاعلي يعرض ملفات المقاولين المعتمدين فقط مع البحث وفلترة المنطقة والتخصص.

### التسجيل المحلي وكلمات المرور

- تُحفظ حسابات العملاء في `bunya-customer-registrations`.
- لا تُحفظ كلمة المرور كنص صريح؛ يُحفظ إثبات SHA-256 محلي للاستخدام التجريبي فقط.
- جلسة الدخول التجريبية تُحفظ في `bunya-local-session` ثم يُوجّه العميل إلى Route `/customer` الموجود مسبقًا.
- كل حقل كلمة مرور أضيف في هذه المهمة يملك Checkbox مستقلًا تحته للإظهار والإخفاء.
- طلبات الاستعادة تُحفظ في `bunya-password-reset-requests` ولا يتم إرسال بريد أو SMS.
- لا تمثل هذه المحاكاة نظام مصادقة آمنًا للإنتاج، ويجب استبدالها مستقبلًا بـ Supabase Auth أو خدمة مصادقة مكافئة.

### بوابة المزودين

- النموذج مقسم إلى بيانات الشركة، الموقع، المنتجات، والتوصيل.
- التصنيفات الأساسية تأتي من بيانات منتجات المنصة الفعلية.
- يدعم تصنيفات مخصصة متعددة، مع منع الفراغ والتكرار والحذف.
- يدعم مناطق توصيل متعددة فقط عندما يختار المزود توفر التوصيل.
- الطلبات تحفظ في `bunya-provider-applications` بحالة `pending` ومع `id` و`createdAt`.
- الطلبات ستظهر مستقبلًا في قسم `طلبات الانضمام للمزودين`، وهو غير مبني ضمن هذه المهمة.

### تحليل روابط Google Maps

الأداة المستقلة `parseGoogleMapsLink` تدعم استخراج الإحداثيات محليًا من:

- `@lat,lng`

الروابط المختصرة من `maps.app.goo.gl` و`goo.gl` تُقبل وتُحفظ دون اختراع إحداثيات. فك إعادة التوجيه واستخراج الإحداثيات منها مؤجل إلى خدمة خادم مستقبلية.

### الخريطة والمسارات

- `/merchant/orders` و`/merchant/orders/[id]`: الطلبات والتفاصيل وانتقالات الحالة المقيدة.

- `/merchant/support`: السياسات والأسئلة الشائعة والتذاكر.
- `/driver`: التحقق من كود التسليم دون إظهاره.

### التدفقات التشغيلية

- عرض السعر: `new → viewed → quoted`، ثم يُحفظ `ProviderQuote` بحالة `pending_customer`. تعديل عرض قائم يتطلب إقرارًا صريحًا.
- الطلب: موافقة العميل التجريبية على العرض تنشئ طلبًا `confirmed`. الانتقال التشغيلي مقيد بالتسلسل: `preparing → ready_for_pickup → assigned_driver → out_for_delivery → delivered → completed`.
- التسليم: يُسنَد الطلب إلى سائق ويُحفظ إثبات SHA-256 فقط للكود. تدخل بوابة السائق رقم الطلب والكود، وتحد المحاولات محليًا، ولا تُغلق مرحلة التسليم دون التطابق.
- المالية: تسليم الطلب ينشئ حركة معلقة، وإكماله يجعل مبلغ الطلب متاحًا. طلب التصفية محلي ولا يتجاوز الرصيد المتاح ولا ينفذ تحويلًا حقيقيًا.

## فصل التوريد الداخلي عن عرض العميل — ملف 015

## 4. قواعد الأعمال المحمية في قاعدة البيانات

- العميل لا يقرأ عروض المزودين أو الاختيار الداخلي؛ يستلم عرض بُنية الموحد فقط.
- قبول العرض يتم عبر `accept_customer_quote` داخل Transaction واحدة: قفل العرض، فحص المالك والصلاحية والمجاميع، إنشاء الطلب والبنود، تحديث الحالات، سجل الحالة وOutbox، مع Idempotency.
- اختيار السعر عبر `select_best_provider_price` للإدارة المخولة فقط، ويشترط الاستهداف، السعر غير المنتهي، الكمية، المنطقة، الموعد، ثم يرتب حسب landed cost.
- الضريبة 15% تحسب أو تستخرج من السعر الشامل بدل تسجيلها صفرًا.
- تأكيد سعر المزود عبر RPC فقط، ويضبط آخر تأكيد والانتهاء بعد 72 ساعة ويسجل confirmation مستقلًا.
- Transition guards لطلب التسعير وعرض المزود والطلب والعرض الموحد والتوصيل والمراحل.
- الطلب مرتبط بعرض بُنية لا بعرض مزود، والتنفيذ الداخلي يحتفظ بالمصدر المختار.
- السائق مرتبط بمزود، ولا يمكن إسناده إلى تنفيذ مزود آخر.
- لا تنتقل التوصيلة إلى `delivered` إلا بعد نجاح كود العميل؛ hash مع salt، حد أقصى 5، وقفل 15 دقيقة.
- فرص المقاول تعرض عبر RPC منزوعة هوية العميل وبيانات التواصل ورابط الخريطة والإحداثيات.
- تعليقات المقاول لا تظهر للعميل إلا بعد مراجعة الإدارة؛ RPC العميل يستبعد أسباب وملاحظات الإدارة الداخلية.
- لا تحذف الفواتير والمدفوعات والحركات والتسويات حذفًا فعليًا.
- العمليات الحساسة تسجل Audit وOutbox؛ وسجل التدقيق غير قابل للتعديل أو الحذف.
- إنشاء الحساب في `auth.users` ينشئ profile فقط؛ لا يثق في role داخل user metadata. تهيئة العميل ومنح بقية الأدوار أو الموافقة عمليات صريحة.

### أخطاء التنفيذ والسلوك التي أثبتتها الاختبارات

1. **Critical DDL:** فهرس على عمود `orders.provider_id` غير موجود.
2. **High Authorization:** `protect_profile_privileged_fields()` كان يمنع `service_role` ويقبل أي Admin بدل Permission محددة.
3. **High RLS/Trigger interaction:** `validate_provider_delivery_assignment()` عمل بصلاحية السائق، فلم ير fulfillment/order عبر RLS ورفض انتقالًا صحيحًا.
4. **High least privilege:** `is_admin()` كان يجعل أي Admin نشط يطابق عشرات السياسات القديمة الواسعة، بما فيها Auditor.
5. **High column-level mutation:** سياسات contractor projects/milestones، notifications، delivery assignments، internal fulfillment وprovider quotes سمحت بتعديل أعمدة أكثر من المطلوب.
6. **High privacy:** العميل كان يستطيع قراءة `provider_quotes` القديمة وعناصرها ومرفقاتها وسجلها، بما يكشف العرض الداخلي للمزود.
7. **Medium runtime:** `audit_sensitive_row()` افترض وجود عمود `id` وفشل تصميميًا مع `platform_settings.setting_key`.
8. **Medium Storage:** تحويل أول مجلد إلى UUID مباشرة كان يستطيع رمي exception لمسار غير صالح.
9. **Medium performance:** وُجدت 4 مفاتيح أجنبية بلا leading index.

أخطاء Fixture مثل تكرار slug seeded، محاولة إدخال قيمة في generated column، أو Timestamp اختبار غير صالح تراجعت داخل Transactions ولم تكن عيوبًا في Migration.

- استبدال `orders_provider_status_idx` بـ`orders_status_created_idx`.

- جعل تأكيد السعر يكتب confirmation واحدًا ويترك Trigger يحدث السعر مرة واحدة، بدل تحديثين متعاقبين.
- تحصين Storage UUID عبر `safe_storage_folder_uuid()` وإرجاع `NULL` للمسار غير الصالح دون exception.
- حماية payload الإشعارات؛ المستلم يغير `read_at` فقط ولا ينشئ إشعارًا ذاتيًا.
- حماية هوية وقيمة مشروع المقاول، ومنع المقاول من اعتماد Milestone أو حذفها، وقصر اعتمادها على العميل.

- حماية Assignment identity/schedule، منع تبديل السائق بعد بدء التوصيل، والتحقق تحت Security Definer آمن.
- فصل Policies المزود للتوصيل إلى Select/Insert/Update، وعدم منحه Delete عامًا.

- حماية آخر Super Admin في حالتي Update وDelete.

| `get_customer_project_comments` | المالك رأى comment معتمدًا | العميل الآخر حصل على 0 | raw table بقي محجوبًا |

الاستثناءات داخل RPCs تراجعت Transactionally؛ لم يبق Idempotency row أو entity جزئية في اختبارات الفشل.

## 8. قواعد الأعمال المختبرة

- انتهاء السعر بعد 72 ساعة، وعدم اختيار السعر المنتهي.
- منع المزود غير المستهدف من الرد.
- حجب هوية المزود عن Bunya customer quote.
- قبول العرض مرتين لا ينشئ طلبين.
- Totals مشتقة ومطابقة للعناصر؛ `total` generated في العرض الموحد.
- منع قفزات order وfulfillment وdelivery غير الصحيحة.
- لا يكتمل delivery بلا كود verified.
- خمس محاولات خاطئة ترفع attempts وتضع `locked_until` وتكتب audit attempts.
- السائق يتبع المزود نفسه، ولا يمكن تغييره بعد بدء الرحلة.
- opportunity projection لا يكشف هوية العميل.
- المقاول لا يعتمد Milestone بنفسه، والمشروع لا يكتمل قبل اعتمادها.
- Audit rows وfinancial rows المحمية لا تُحذف عبر المسارات المصرح بها.
- آخر Super Admin لا يُعطل أو يُحذف.
- `platform_settings` لا يعدلها Operations Admin بلا Permission، ويكتب Super Admin Audit صحيحًا بـ`setting_key`.

## 10. الأداء

- كل Foreign Key لديه leading index مناسب.
- `orders` الخاص بالعميل استخدم `orders_customer_created_idx` مباشرة.
- contractor opportunity query استخدم `contractor_opportunities_profile_status_deadline_idx` مباشرة.
- provider pricing response query استخدم Bitmap Index على `provider_pricing_responses_provider_created_idx` ثم Sort عند التصفية بالحالة؛ هذه ملاحظة Low، ويمكن إضافة `(provider_id, status, created_at desc)` فقط بعد قياس workload الحقيقي.
- لم تظهر sequential scans مقلقة في الاستعلامات الأساسية المختبرة، مع التنبيه أن Fixtures صغيرة وليست benchmark إنتاجيًا.

## 11. الأخطاء المتبقية حسب الشدة

### Critical

**0**.

### High

**0** معروفة في النطاق المختبر.

### Medium

1. لم يُختبر Supabase Auth/Storage API أو signed URLs/MIME rejection فعليًا؛ PostgreSQL Stubs لا تعوض Dry Run على Supabase Local/Staging.

## 12. Breaking Changes المتوقعة

- أي كود كان يعتمد `is_admin()` للدلالة على «أي موظف إدارة» يجب أن يستخدم Permission محددة؛ `is_admin()` الآن Super Admin فقط.
- العميل لم يعد يستطيع قراءة `provider_quotes` القديمة بأي شكل.
- المستلم لا يستطيع تعديل notification payload؛ `read_at` فقط.
- المقاول لا يستطيع اعتماد milestones أو تعديل القيمة/هوية المشروع.
- المزود لا يستطيع تعديل القيمة أو الربط المرجعي للـfulfillment، ولا relink للـprovider quote.
- file registry soft delete غير قابل للعكس من client.
- لا يجوز تشغيل الملف فوق بيئة طبقت 001 سابقة؛ هذا Baseline لقاعدة فارغة فقط.
